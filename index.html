Madness Oracle Pro

64-team betting workbench

Overview

A real 64-team March Madness betting and bracket workbench.

Yes — the tournament has 64 teams in the main bracket. This version models the full field, simulates all six rounds, and turns probabilities into betting edges so you can hunt for mispriced lines instead of pretending any app can be 100% right.

How to make the best bets

  • 1. Click Load official 2026 field after a reset or if you want a fresh bracket.
  • 2. Click Sync results, odds + news to refresh the live round and sportsbook prices.
  • 3. Click Generate + apply AI prefill to load coach, injury, form, experience, and continuity context into Team Lab and the model.
  • 4. Click Generate AI betting settings to tune Kelly, edge floor, confidence floor, and hedge style.
  • 5. Run 25,000 simulations for a quick pass or 100,000 for a cleaner final board.
  • 6. Go to the Betting workspace and use the top recommendation cards there.

If you apply live odds, news, AI prefill, evolved weights, or evolved strategy, rerun the simulations before trusting the board. Backtest, score replay, evolution lab, and strategy lab are optional deeper checks, not required for the daily workflow.

Architecture + privacy

What this app is — and what it is not

Today this can run two ways: as a local Rust + Axum web app, or as a desktop-ready Tauri shell using the same SQLite-backed state model.

Private by default

Predictions stay on-device unless you explicitly export them. Saves are written to a local SQLite file through the Rust app.

No cloud backend

There is a local API now, but no cloud sync and no remote database collecting picks. That keeps the prototype private by default, but still means no cross-device syncing.

SQLite local DB

SQLite is now the persistence layer for the prototype, which gives you a real local database without forcing users to upload private picks.

Tauri status

This repo now includes Tauri desktop scaffolding so the next packaging step is an installable desktop app instead of a browser-only workflow.

Bracket engine

Region-by-region simulation path

Completed NCAA games are locked in first, then the unresolved parts of each region are simulated forward through the Final Four and title game.

Heavy runs

Push the optimizer deeper without freezing the app

Larger populations, longer generations, and multi-start search run as background jobs. Kick them off here, then keep using the rest of the dashboard while they work.

Backtest

How the system did on prior years

Historical results use archived NCAA bracket pages. Where archived NCAA team-stat feeds are unavailable, the backtest falls back to the model's seed-based proxy ratings.

Score replay

No-lookahead score grading on prior years

Each archived game is frozen before grading: the system projects winner probability, margin, total, and team scores using only the pregame field lookup for that season, then compares that snapshot to the actual final.

Evolution lab

Learn stable meta-signals without chasing noise

This search repeatedly mutates the model weights on archived tournaments, checks them with leave-one-year-out folds, and shrinks the answer back toward the default profile before suggesting anything for the live season.

Strategy lab

Learn how this model should actually bet

This search keeps the base model fixed and evolves a paper-trading profile instead: edge threshold, Kelly scaling, stake caps, open-risk caps, confidence gating, and hedge style.

Research stance

What current academic-style research informed

Market-aware

Good betting workflows compare model probabilities against prices. Research on betting efficiency suggests markets are tough, so this app focuses on edge detection, vig removal, and disciplined uncertainty.

Full bracket, not toy demo

Tournament probabilities change with path dependence. Full-field Monte Carlo simulation captures how one region path changes Final Four and title equity later.

Four Factors + efficiency

Adjusted efficiency plus the Four Factors remain strong baseline signals, which is why they anchor the model instead of being bolted on as afterthoughts.

Human override friendly

The model is editable because real workflows benefit from expert overrides for injuries, lineup changes, coaching edges, experience, and market dislocations that static academic models can miss.
